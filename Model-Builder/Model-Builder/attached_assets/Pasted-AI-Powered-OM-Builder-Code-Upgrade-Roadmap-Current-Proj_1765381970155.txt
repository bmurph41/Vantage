AI-Powered OM Builder – Code Upgrade Roadmap
Current Project Architecture and Features

The existing OM Builder project is a full-stack TypeScript application consisting of a React front-end and an Express/Node back-end. The client is built with Vite and React (TypeScript), organized into pages (e.g. om-builder, om-export, project-oms) and components. Key front-end components include an OM editing canvas, side panels (for AI assistance, data binding, branding, etc.), and a block renderer for various content blocks. The back-end uses Express with a RESTful API and PostgreSQL via Drizzle ORM. Data models are defined for users, offer memoranda (OMs), OM pages, OM content blocks, OM templates, and datasets (e.g. for underwriting Excel data). For example, the omBlocks table stores each slide block’s type (text, chart, image, etc.), content JSON, data bindings, and style metadata. The API endpoints handle creating and managing OMs, pages, blocks, templates, and uploading datasets (Excel/CSV). An AI service module integrates OpenAI’s API for text generation, with endpoints to generate content (e.g. executive summaries, highlights), improve existing text, and suggest slide layouts. The project already supports basic drag-and-drop reordering of items via @dnd-kit and uses Recharts for charts in slides.

Overall, the codebase is modular and follows good practices (e.g. a storage layer abstracts database operations, Zod schemas validate input). This provides a solid foundation to build upon. The goal is to preserve this existing code and structure as much as possible, while implementing the full feature roadmap for an AI-powered OM (Offering Memorandum) builder.

1. Preserving and Upgrading Existing Code

We will retain the current project structure (separating client and server code, using Drizzle ORM, React components, etc.) and reuse code where possible, incrementally upgrading it to support new features. Key existing components (both front-end and back-end) should be preserved and enhanced rather than rewritten from scratch:

Front-end: Keep the React/Vite setup and the component-based UI. Continue using Tailwind CSS (and Radix UI primitives) for consistent styling. Reuse the current OM editing page and side panels, enhancing them with new capabilities (drag-and-drop layout, AI integration UI, etc.). For example, maintain the current block rendering logic (text, images, charts) but extend it with new block types and improved styling.

Back-end: Keep the Express server and routing structure, as well as the Drizzle ORM data access. Continue using the existing Postgres database schema for OMs, pages, blocks, etc., extending it where needed (e.g. adding fields or tables for new features like roles or organizations). Reuse utility code (logging, error handling, etc.) and the OpenAI integration in ai-service.ts for text generation.

While preserving code, we will refactor for clarity and modularity. For instance, the single routes.ts can be split into logical route modules (e.g. omRoutes, templateRoutes, dataRoutes, aiRoutes, authRoutes) mounted on the Express app. This will organize the growing number of endpoints for maintainability. We’ll also add comments and docstrings throughout the code to improve readability and help future developers. Any sections of code flagged as placeholders or incomplete will be fleshed out (for example, if user authentication or certain template logic was left stubbed, we will implement those fully now). The upgraded codebase should remain modular, maintainable, and Replit-friendly, meaning it should run within the Replit environment with minimal configuration changes.

2. Integrating Full OM Builder Roadmap Features

Now we integrate the features from the roadmap into the existing codebase. Each sub-feature from the roadmap will be implemented as described below:

Drag-and-Drop Slide Building (Freeform Layout)

Add support for freeform drag-and-drop placement and resizing of slide content blocks, moving beyond simple list reordering. We will integrate a library such as React Grid Layout to achieve a draggable, resizable grid system on each slide. React Grid Layout provides “a draggable and resizable grid layout with responsive breakpoints, for React”
github.com
, which fits our need for a slide canvas where users can position text boxes, images, charts, etc., in a flexible layout.

Implementation: Create or refactor the slide canvas component (currently canvas.tsx) into a SlideCanvas component that uses <ResponsiveGridLayout> (from react-grid-layout) or a similar grid. Each OM block will correspond to a grid item that can be dragged and resized. We will need to store layout coordinates (x, y positions and width/height in grid units) for each block. We can utilize the existing omPages.layout JSON field to store an array of layout info or map of block IDs to layout properties. Alternatively, extend the omBlocks schema to include fields for position and size. The front-end should capture layout changes (on drag/resize stop) and send updates to the back-end to save. This ensures that when a user reopens the project, blocks appear in their saved positions. We will use the library’s callbacks (e.g. onLayoutChange) to update state and call an API (e.g. PATCH /api/blocks/:id with new coordinates).

On the UI side, ensure the drag-and-drop interface is intuitive: show a grid background or guidelines, and maybe a zoom or snap-to-grid for alignment. The SlideCanvas should handle different block types gracefully – for example, text blocks can resize vertically, image blocks maintain aspect ratio, etc. If using Interact.js instead (another option), the concept is similar: make elements draggable and resizable and update their style/position in state. But React Grid Layout is likely simpler for a grid-based slide design. We will choose one (React Grid Layout for concreteness) and include it in package.json if not already there.

Back-end: No separate endpoints are needed specifically for drag/drop, since block position can be updated via the existing block update route (PATCH /api/blocks/:id). We will ensure this route can handle layout updates (x, y, w, h) and store them (perhaps in omBlocks.style or omPages.layout). The real-time collaborative movement (if multiple users) will be addressed later with Yjs, but initially drag/drop is single-user.

JSON Templating for Common Slides

Implement a templating system to allow quick creation of common OM slides (Investment Highlights, Market Overview, Executive Summary, etc.) from pre-defined templates. The project already has an om_templates table and endpoints for templates, so we will leverage that. Each template will consist of JSON data defining a slide’s structure – e.g. a title and a set of blocks with placeholder content.

Template Definition: We will create JSON schemas or objects for key slide types:

Investment Highlights – e.g. a slide with a title “Investment Highlights” and a bullet-point list block for 5-6 key highlights.

Market Overview – e.g. a slide with a title and a mix of text and image (such as a map or chart of the market).

Executive Summary – e.g. a slide with a title and a multi-paragraph text block summarizing the deal.

Financials Summary – e.g. a table or key metrics block with summary financial data.

(Additional templates: Property Overview, Area Map, Team info, etc., as relevant.)

These template definitions can be stored as JSON files or directly in code and inserted into the database as global templates on startup. We will include sample template JSON structures for a couple of slide types in the codebase (for example, in a seeding script or as part of the template module). Each template JSON will list the blocks (with type, default content, style, etc.) and possibly layout info.

Template Application: In the front-end UI, provide a way for the user to choose a template when adding a new slide. For instance, an “Add Slide” button could open a template selector (with template names like “Investment Highlights”, “Market Overview”, etc.). On selection, the client calls an endpoint (e.g. POST /api/oms/{omId}/pages with a templateId or template data), and the server will populate a new page and its blocks according to the template’s JSON (templateData). The new slide appears in the editor pre-filled with placeholder text and structure, which the user can then customize.

We will implement the template insertion logic on the server side using the om_templates.templateData. For example, if the templateData JSON contains an array of blocks, the server will create a new page record (with given title) and create block records for each block in the template (associating them to the new page). This logic can be added to an endpoint like POST /api/templates/:templateId/apply or integrated into the existing POST /api/pages (if a templateId is provided in the request body).

We will also allow saving custom templates. When a user has designed a slide, they should be able to save it as a template for reuse. We’ll add a route like POST /api/templates/save that takes a page ID and creates a new template entry by copying its blocks and content into templateData. The ownerType and ownerId fields of templates will govern visibility: e.g. if saved at user-level or org-level. (The schema already has ownerType with values global | org | user to support this.)

Internally, Zod schemas will be used to ensure template JSON is valid. We might create Zod definitions for a “Template” structure (list of blocks with required fields) to validate incoming template saves. This prevents invalid templates from being stored.

For Suggest Layout: integrate a feature where the user can describe the content of a slide or the points they want to cover, and the AI suggests an optimal layout or template. The /api/ai/suggest-layout endpoint returns a suggested template name, reasoning, and a list of recommended blocks with content. We will use this in the UI as a “AI Layout Suggestion” feature. For example, if the user is unsure how to present a certain data, they can input a brief description (e.g. “occupancy history and forecast”) and the AI might suggest a chart plus bullet points, etc., which we can then automatically assemble using available block components. This tightly integrates AI into the slide-building process.

OpenAI API Config: Ensure the app uses environment variables for OpenAI API Key and endpoint. The code expects AI_INTEGRATIONS_OPENAI_API_KEY and possibly a base URL. In the README, instruct the developer to set up an OpenAI API key (for GPT-4 access) and configure these env vars. We will upgrade the README to explain this (see Deliverables section).

Throughout, focus on GPT-4 usage to generate high-quality, context-aware text. The prompts should be tuned for commercial real estate language. The system prompt in ai-service ensures the AI writes in a professional OM style (this is already in place, and we will keep it or enhance it as needed). With these AI features, the OM builder will significantly speed up content creation (e.g. generate a polished Executive Summary at the click of a button, covering major selling points of the property in a style similar to real OMs).

We will enforce that only authorized users can access or modify an OM or template: e.g. a user can only fetch OMs for their own project/org. The frontend should be adjusted to handle auth: provide a Login page (simple form to authenticate). Role-based permission ensures that content can be shared appropriately: user-level content (only visible to creator), team/org-level (visible to all in org), global (available to all users of the system, e.g. sample templates). This aligns with how templates are intended to work (ownerType field) and we’ll propagate the same concept to projects and OMs.
