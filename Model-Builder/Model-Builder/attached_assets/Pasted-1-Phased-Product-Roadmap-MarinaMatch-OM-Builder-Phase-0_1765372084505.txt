1. Phased Product Roadmap – MarinaMatch OM Builder
Phase 0 – Foundation & Alignment

Goal: Make sure the OM Builder plugs cleanly into your existing stack and data model without breaking anything.

Objectives:

Confirm:

Where “Projects/Deals” live (table & route names).

Where underwriting, sales comps, rent comps, and demographics are stored and how they’re exposed via APIs.

Decide:

URL structure for OM Builder routes.

Where to store OM-related schema (Drizzle) and frontend modules.

Key decisions:

Use existing project/deal entity as the “parent” of every OM:

Each OM always belongs to a single project.

Create a data façade for OM consumption:

/api/om-data/underwriting/:projectId

/api/om-data/sales-comps/:projectId

/api/om-data/rent-comps/:projectId

/api/om-data/market/:projectId

/api/om-data/demographics/:projectId

Phase 1 – Core OM Editor (No AI, Minimal Data Binding)

Goal: A fully functional OM editor wired to projects: OMs → pages → blocks, with CRUD + drag-and-drop, and the ability to export a simple-but-professional PDF.

User Stories

As a user, I can:

Create an OM for a project.

Add/remove/reorder pages within that OM.

Add/remove/reorder content blocks (text, KPI, chart placeholder, table placeholder, image placeholder, map placeholder).

Click into a block to edit its content and style.

Save and re-open OMs later.

Export as a decent PDF.

Backend (Phase 1)

Drizzle schema (new tables)

Create new schema file, e.g. /backend/db/schema/om.ts with:

oms

om_pages

om_blocks

om_templates (basic structure, but template features will really come in Phase 2–3)

We already defined fields; Phase 1 uses:

oms: id, project_id, name, status, version, settings, created_by, updated_by, timestamps.

om_pages: id, om_id, title, order_index, layout, timestamps.

om_blocks: id, page_id, type, order_index, content, data_binding, style, ai_metadata, timestamps.

om_templates: id, owner_type, owner_id, name, scope, category, template_data, created_by, timestamps.

Even if templates are underused in P1, creating the table now avoids schema churn.

Pages:

Data binding + AI can be placeholders for now (data_binding and ai_metadata default to null/empty).

Goal: Turn the static editor into a data-aware, AI-assisted OM builder.

User Stories

Backend (Phase 2)

OM Data Facade Endpoints

Implement:

GET /api/om-data/underwriting/:projectId

GET /api/om-data/sales-comps/:projectId

GET /api/om-data/rent-comps/:projectId

GET /api/om-data/market/:projectId

GET /api/om-data/demographics/:projectId

Each returns:

{
  metrics: Record<string, number | string>,
  series: Array<{
    id: string;
    label: string;
    data: Array<{ x: string | number | Date; y: number }>;
  }>,
  tables: Array<{
    id: string;
    label: string;
    columns: { id: string; label: string }[];
    rows: Record<string, any>[];
  }>
}


Map your existing data into this generic shape. For example:

Underwriting metrics: NOI_Year1, CapRateEntry, PurchasePrice, etc.

Series: noiByYear, revenueByYear.

Tables: T-12 P&L summary.

2. Implementation Notes / Guardrails

Keep OM Builder decoupled from internal schema details via the /api/om-data/* facade. That way, if your underwriting model evolves, you only adjust that layer.

Keep templates data-agnostic:

Use generic binding keys (“primary_noi_series”, “primary_revenue_mix”), then map those to specific metrics per project behind the scenes.

Do not overcomplicate layout in P1. Stacked blocks in a single column are fine; you can add 2-column layouts later.

A. DATABASE – DRIZZLE SCHEMA & MIGRATIONS

1. Create a new Drizzle schema file, e.g. `/backend/db/schema/om.ts`, respecting current project naming patterns (import paths, schema registries, etc.).

2. Define the following tables (adjust naming to match repo conventions: snake_case or camelCase):

TABLE: oms
- id: primary key (UUID or whatever is standard).
- project_id: FK → existing projects/deals table, NOT NULL.
- name: string.
- status: enum or constrained text: 'draft' | 'published' | 'archived'.
- version: integer, default 1.
- settings: JSON, nullable.
- created_by: FK to users table, if available; nullable if needed.
- updated_by: FK to users.
- created_at, updated_at: timestamps with default now().

TABLE: om_pages
- id: primary key.
- om_id: FK → oms.id, NOT NULL.
- title: string.
- order_index: integer – page ordering within OM.
- layout: JSON (page-level layout/branding options, can be minimal for now).
- created_at, updated_at.

TABLE: om_blocks
- id: primary key.
- page_id: FK → om_pages.id, NOT NULL.
- type: text/enum: 'text' | 'chart' | 'table' | 'kpi' | 'image' | 'map' | 'custom'.
- order_index: integer – vertical order on the page (for now).
- content: JSON – block-type-specific content. For Phase 1:
  - text: `{ markdown: string }`
  - kpi: `{ label: string, format?: 'currency'|'percent'|'number', fallbackText?: string }`
  - chart/table/image/map: placeholder objects with basic metadata fields.
- data_binding: JSON, nullable – leave this for Phase 2.
- style: JSON, nullable – for block appearance (alignment, typography, card style).
- ai_metadata: JSON, nullable – reserved for AI usage later.
- created_at, updated_at.

TABLE: om_templates
- id: primary key.
- owner_type: 'global' | 'org' | 'user'.
- owner_id: nullable FK.
- name: string.
- scope: 'block' | 'page' | 'om'.
- category: string.
- template_data: JSON.
- created_by, created_at, updated_at.

3. Create migration(s) for these tables, following existing migration infrastructure. Ensure foreign keys match your current schema.

B. BACKEND – EXPRESS ROUTES & CONTROLLERS

Create a new router module, for example `/backend/routes/omBuilder.ts` or `/backend/routes/om/index.ts`, and wire it into the main Express app under a prefix like `/api`.

Implement the following REST endpoints, using TypeScript and Drizzle queries:

1. OM ROUTES

- `GET /api/projects/:projectId/oms`
  - Validate that the authenticated user has access to this project.
  - Return list of OMs for that project with: id, name, status, version, timestamps.

- `POST /api/projects/:projectId/oms`
  - Body: `{ name: string, status?: 'draft'|'published'|'archived', settings?: any }`.
  - Create an OM associated with the project.
  - Default status = 'draft', version = 1.
  - Return created OM.

- `GET /api/oms/:omId`
  - Return:
    ```json
    {
      "om": { ... },
      "pages": [
        { ...page, "blocks": [ ...blocks ] },
        ...
      ]
    }
    ```
  - Use joins or multiple queries to assemble this tree.

- `PATCH /api/oms/:omId`
  - Allow updates to name, status, settings.
  - Update updated_by and updated_at.

- `DELETE /api/oms/:omId`
  - Follow existing pattern (soft delete via deleted_at or hard delete). Ensure pages + blocks are also removed or cascade appropriately.

2. PAGE ROUTES

- `POST /api/oms/:omId/pages`
  - Body: `{ title: string, order_index?: number, layout?: any }`.
  - If order_index is undefined, append to end (largest order_index + 1).
  - Return created page.

- `PATCH /api/om-pages/:pageId`
  - Body can include `{ title?, order_index?, layout? }`.
  - Persist changes.

- `PATCH /api/oms/:omId/pages/reorder`
  - Body: `{ pageOrder: { pageId: string; order_index: number }[] }`.
  - Batch update order_index for given pages.

- `DELETE /api/om-pages/:pageId`
  - Delete page and its blocks (cascading or manual).

3. BLOCK ROUTES

- `POST /api/om-pages/:pageId/blocks`
  - Body:
    ```ts
    {
      type: 'text' | 'chart' | 'table' | 'kpi' | 'image' | 'map' | 'custom';
      order_index?: number;
      content?: any;
      style?: any;
      data_binding?: any; // can be empty for now
    }
    ```
  - If order_index is undefined, append to end.
  - Return created block.

- `PATCH /api/om-blocks/:blockId`
  - Partial update for: order_index, content, data_binding, style.

- `PATCH /api/om-pages/:pageId/blocks/reorder`
  - Body: `{ blockOrder: { blockId: string; order_index: number }[] }`.

- `DELETE /api/om-blocks/:blockId`
  - Delete block.

4. PERMISSIONS

- Confirm user has access to the project the OM belongs to.
